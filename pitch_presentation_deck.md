## 📌 Slide 3: Elastic Usage — The Core Engine
> **Presenter Quote:** *"Elastic is the brain of Shrayak. We utilize Elastic Cloud Serverless in four distinct ways, maximizing search relevance, location intelligence, and operational analytics."*

1.  **Hybrid RAG Search with Reciprocal Rank Fusion (RRF)**:
    *   Combines **Semantic Search** (Cosine similarity on 768-dimensional dense vectors generated by `text-embedding-004`) with **Keyword Search** (BM25 on statutes, tags, and titles).
    *   *Why this matters for judges:* Workers ask questions in highly variable, colloquial Hindi/English transliterations ("*bocw card kaise banaye*", "*kitna paisa milega*"). Vector search captures the conceptual intent, while BM25 guarantees that exact section numbers (e.g. "Section 14 of Minimum Wages Act") are retrieved and cited.
2.  **Geo-Location Routing (`geo_point` & `geo_distance`)**:
    *   The `delhi_labour_offices` index maps all district labor offices and welfare board HQs with physical GPS coordinates (`location: { lat, lon }`).
    *   When a worker types their pincode (e.g. `110092`), the system resolves the GPS coordinate and runs a `geo_distance` filter to find the **nearest office, helpline, and closest Metro station** in sub-milliseconds.
3.  **Real-Time Analytics & Aggregations**:
    *   Queries the `delhi_workers` index on the fly using Elastic aggregations to calculate average wages paid and underpayment percentages across Delhi districts, powering the real-time dashboard.
4.  **Kibana-Ready Observability Logging**:
    *   Every single transaction is logged to `telemetry_logs` index to monitor user latency, retrieval scores, and error rates.

## 📌 Slide 4: Real-Time Data & Data Effort
> **Presenter Quote:** *"We built four live data feeds into the system, ensuring that neither the LLM nor the worker is looking at stale data."*

*   **Live Official Wages**: Evaluates worker wage rates against live values retrieved dynamically from the `delhi_wages_2026` Elastic index on boot, calculating compliance instantly.
*   **Worker Registry Integration**: Simulates a live registry of daily wage workers. When you search a worker by name or UAN, it tells you if they are registered under the BOCW Board and flags if their daily wage is compliant with the latest government notification.
*   **Labour PIB News Feed**: Ingests real-time labor circulars from the Press Information Bureau (PIB) RSS feed, indexes them into `delhi_news_feed` (Serverless-compatible mapping), and serves them in a scrollable sidebar.
*   **eShram Live Counter**: Shows real-time national registration counts in the top header.

## 📌 Slide 5: Observability & Observability Telemetry
> **Presenter Quote:** *"Observability is built-in, not bolted on. We monitor every step of the agent's journey."*

*   **Telemetry Logs Index**: We log latency (Embedding generation time vs. Elasticsearch query time vs. LLM generation time) to analyze bottleneck areas.
*   **Retrieval Health**: We monitor the RRF scores of documents. If a query yields low scores, we flag it in our logging system as a "grounding gap" to prompt further data ingestion.
*   **Audit Trail**: Logs success/failure flags and identifies which safety rules (like PII detection) were triggered.

## 📌 Slide 6: Zero-Trust Security & Resiliency
> **Presenter Quote:** *"Workers trust us with sensitive information. We implement a Zero-Trust architecture to safeguard their identity."*

*   **Edge PII Stripper**: Runs a regex-based sanitization engine on the query *before* it reaches external APIs (like Gemini Embeddings). Standard PII (Aadhaar cards, phone numbers, PAN, email) is stripped at the server edge.
*   **Strict Dynamic Mappings**: We configure Elastic indices with `dynamic: strict`. This blocks indexing of unexpected fields, preventing data pollution or structural injection attacks.
*   **Zero Hardcoded Credentials**: Every credential (Elastic Serverless tokens, API keys) resides securely in `.env` (git-ignored) and runs via runtime environment variables.
*   **Robust Offline Fallback Engine**: If Gemini is down or throttled, our rule-based local knowledge processor intercepts the call, fetches the grounding document from Elastic, and returns a template-based answer in Hindi/English so the chat **never crashes**.

# 📝 Presenter's Cheat Sheet: Quick Q&A Prep

#### Q1: "Why did you use Hybrid Search (RRF) instead of pure kNN?"
*   *Answer:* "Pure kNN vector search is great for semantic meaning but terrible for exact numbers or specific statutes (like searching 'Section 12 of BOCW Act'). BM25 keyword search is precise for text matching. By combining both via Reciprocal Rank Fusion, we ensure we capture both the worker's semantic intent and exact statutory references."

#### Q2: "How is security handled in this AI agent?"
*   *Answer:* "We have a strict Zero-Trust approach. We run a PII stripper at our API gateway so that sensitive worker details (like Aadhaar card numbers, phone numbers, or contractor names) are scrubbed before they get sent to external LLMs. Furthermore, our Elasticsearch indexes use strict mapping rules to prevent dynamic field injection."

#### Q3: "What happens if the LLM API is down during a critical worker request?"
*   *Answer:* "We engineered a local fallback engine. If the Gemini API fails, our system automatically parses the query intent, pulls the correct grounding context from Elasticsearch, and populates a localized expert template in the worker's selected language. The worker gets their answer, the office location, and helplines immediately without any downtime."
